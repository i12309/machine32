Нужно делать рефакторинг кода по переходу на новый тип экрана из библиотек `screenLIB` и `ScreenUI`.

Будем постепенно замещать старые страницы новыми и попутно допиливать библиотеки до полного соответствия по методам и функционалу.

---

Текущее состояние рефакторинга экрана

Цель
- Уйти от старого `Nextion/Page`-слоя к новому экранному слою на `screenLIB + ScreenUI`.
- Замещать страницы постепенно, без единовременного переписывания всего UI.
- Сначала поднять стабильный boot-flow и базовые страницы `Load` и `Main`.

Что уже сделано
- Добавлен фасад `src/Screen/Screen.h/.cpp`.
- `Boot` больше не инициализирует `Nextion` напрямую, а работает через `Screen`.
- `setStatus` и `setStatusFail` упрощены: вывод только в лог.
- В основном цикле `App::process()` добавлен `Screen::tick()`.
- Удалены старые следы `native/screenprep`: `src/Screen/BOOT_example.cpp`, `src/backend_main.cpp`.
- Новый `Load` переведен на generated ids из `ScreenUI`.
- Старый `main_menu.h` заменен на `src/Screen/main.h`, который опирается на новый `screenui::MainBase`.
- В `screenLIB` host runtime добавлен внешний observer событий, чтобы приложение могло получать service-сообщения (`hello`, `device_info`) независимо от страниц.
- `Screen` теперь умеет:
  - отправлять `requestDeviceInfo()`;
  - принимать `hello/device_info`;
  - сохранять `fw_version`, `ui_version`;
  - обновлять текст версии на `Load`, если ответ пришел уже после показа страницы.

Что еще не закончено
- `screen32` пока отдает тестовые значения `DeviceInfo`:
  - `fw_version = "screen32-frontend"`
  - `ui_version = "eez-lvgl"`
- Нормальный version-layer на стороне `screen32` еще не оформлен.
- Переход `Load -> Main` пока не сделан как окончательный механизм.
- Нет общей схемы page-controller/navigation-policy для новых страниц.

Ответы на вопросы

1. Почему вызывается `_runtime.screens().requestDeviceInfo(1);`
- `1` здесь это `request_id`.
- Это не "версия 1" и не special mode.
- Это просто идентификатор запроса, чтобы можно было при желании различать ответы на разные service-запросы.
- Сейчас он выбран как фиксированное значение-заглушка.
- Технически можно было бы передавать и `0`, если корреляция ответов пока не нужна.
- Для текущей логики правильнее либо:
  - использовать `0`, если request/response не различаем;
  - либо завести в `Screen` свой счетчик request id, если будем делать несколько service-запросов.
- Рекомендация: заменить `1` на именованную константу или счетчик, чтобы не было магического числа.

2. Какой механизм перехода в `Main` предлагается

Старый подход
- В старом UI каждая страница сама смотрит на `App::state()->type()` и решает, что делать.
- Пример из `pLoad`:
  - при входе в `IDLE` вызывается `n_Idle()`
  - `_setFunc(__func__)` гарантирует однократный запуск
  - затем вызывается `pMain::getInstance().show()`
- Плюсы:
  - просто;
  - переходы явно лежат рядом со страницей;
  - хорошо читается для небольшого проекта.
- Минусы:
  - логика размазывается по страницам;
  - страницы начинают знать про state-machine;
  - сложнее централизованно контролировать переходы;
  - service-события и навигация начинают смешиваться.

Предлагаемый новый подход
- Сохранить идею "переход по состоянию", но вынести ее из конкретных page-классов в отдельный навигационный слой.
- Страница должна отвечать за:
  - отображение;
  - реакции на кнопки/ввод;
  - локальное поведение.
- А решение "какую страницу показать при каком `State`" должно жить в одном месте.

Рекомендуемая схема
- Ввести в `Screen` понятие активной page-policy/navigation-policy.
- На каждом `tick()` или по событию смены state экранный слой сравнивает:
  - текущий `App::state()->type()`
  - последнюю обработанную state
  - текущую активную страницу
- Если state изменилась, навигационный слой однократно выполняет нужный переход.

Минимальный вариант без большой архитектуры
- Добавить в `Screen` поля:
  - `lastHandledState`
  - `currentPageKind`
- Добавить метод вроде `syncWithAppState()`.
- В нем делать централизованную таблицу переходов:
  - `BOOT -> showLoad()`
  - `IDLE -> showMain()`
  - позже `SERVICE -> showService()`, и т.д.
- Вызывать это централизованно из `Screen::tick()`.

Почему это лучше
- Однократность перехода обеспечивается централизованно сравнением `lastHandledState`.
- Страницы не обязаны знать про `App::state()`.
- Логика навигации читается в одном месте.
- Старую идею "реакция на state" мы не теряем, но делаем ее чище.

Итоговая рекомендация
- Для нового экранного слоя не переносить старую схему `switch(App::state()->type())` в каждую страницу.
- Вместо этого сделать централизованный state-to-page router внутри `Screen`.
- Если нужна однократность, решать ее не через `_setFunc(__func__)`, а через:
  - хранение `lastHandledState`;
  - проверку, изменилась ли state;
  - вызов перехода только в момент изменения.

Ближайший следующий шаг
- Убрать магическое число из `requestDeviceInfo(1)`.
- Добавить в `Screen` централизованный router:
  - `BOOT => Load`
  - `IDLE => Main`
