# AGENT.md для `machine32`

Этот файл фиксирует архитектурные правила проекта и рабочие договоренности.

## 1. Назначение проекта

Прошивка для ESP32 (PlatformIO + Arduino) для полиграфического станка с интерфейсом Nextion.

Ключевые части:
- управление моторами (`PAPER`, `TABLE`, `GUILLOTINE`, `THROW`) через FastAccelStepper;
- работа с датчиками, энкодером и муфтой через GPIO и MCP23017;
- конечный автомат состояний и планировщик шагов;
- UI на Nextion;
- HTTP, MQTT и OTA;
- хранение `config/data/stats` в LittleFS, служебных данных в NVS.

## 2. Точка входа

Файл: `src/main.cpp`

- `setup()`:
  - `Log::init()`
  - `State::init()`
  - `Trigger::init()`
- `loop()`:
  - `Page::process()`
  - `State::process()`
  - `Service::process()`

## 3. Главные подсистемы

### 3.1 Core/Config/Data

Файлы: `src/Core.h`, `src/config.*`, `src/Data.*`

- `Core::config` работает с `config.json`.
- `Core::data` работает с `data.json`.
- `Core::settings` парсится из `config.settings`.
- `Data` хранит runtime-параметры процесса.

### 3.2 Состояния и план

Файлы: `src/State/State.h`, `src/State/PlanManager.*`

- Один активный `State*`.
- Переходы через `Factory(...)`.
- `PlanManager` хранит шаги, состояния, действия и параметры.
- `ACTION` исполняет callback.
- `DONE` возвращает управление в вызывающий контекст.

### 3.3 BOOT

Файл: `src/State/Device/Boot.h`

BOOT выполняет инициализацию Nextion, файловой системы, NVS, config, Wi-Fi, OTA, HTTP, MQTT, data, profiles, stats и registry.

### 3.4 Реестр железа

Файл: `src/Controller/Registry.h`

`Registry` динамически создает устройства по `config.device`.
`Registry::Context` хранит общие ссылки на устройства для состояний.

### 3.5 Триггеры

- `Trigger` — polling.
- `McpTrigger` — IRQ от MCP.

Тяжелая логика только вне ISR.

### 3.6 Scene

Файл: `src/State/Scene.*`

`Scene` — единый фасад рабочих действий механизмов.

Правило: в новых состояниях и UI сначала использовать `Scene`, не дублировать low-level управление моторами.

## 4. Multi-Machine Rules

- Тип станка и его возможности определяются только из `config`.
- Нельзя жестко зашивать возможности конкретного станка без опоры на `config`.
- Логика должна учитывать фактически доступные механизмы текущего типа.

## 5. Сервисы

- HTTP: `src/Service/HServer.*`
- MQTT: `src/Service/MQTTc.cpp`
- Сервисный цикл: `src/Service/Service.h`

`Service::process()` должен выполняться только в безопасных состояниях.

## 6. Хранилище

- LittleFS: `/config.json`, `/data.json`, `/stats.json`
- NVS: `boot`, `wifi`, `licence`

## 7. Сборка

Файл: `platformio.ini`

- `env:esp32dev`
- `framework = arduino`
- `board_build.cxx_std = 17`

## 8. Критичные правила изменений

1. Не делать I2C-операции в ISR.
2. Для движения использовать `IStepper` и `Scene`.
3. Для новых сценариев использовать `PlanManager`.
4. При изменении схемы `config` синхронизировать дефолты, загрузку и UI.
5. Учитывать offline MCP: часть устройств может быть недоступна.

## 9. Чеклист перед завершением задач

- сборка проходит для `env:esp32dev`;
- Boot-переходы корректны;
- не сломаны `Registry::init()` и `Registry::Context::load()`;
- не сломаны `Scene::*Stop()`;
- проверены триггеры;
- проверены чтение и сохранение `/config` и `/data`.

## 10. Главное обязательное правило по языку и кодировке

Это правило обязательно для каждой новой сессии и для любой правки без исключений.

- Никогда не ломать русские символы, кириллицу и кодировку в существующих файлах.
- Все комментарии в коде, пояснения в документах и ответы ассистента писать только на русском языке.
- Все новые и обновляемые `README.md` писать только на русском языке.
- При редактировании текстовых файлов использовать UTF-8 и проверять, что после сохранения текст читается в IDE без ручной перекодировки.
- Если файл уже в другой кодировке, нельзя молча перекодировать его вместе с функциональными правками. Смена кодировки должна быть осознанной и проверяемой.
- Запрещено оставлять mojibake вида `Р...`, `С...`, `Ð...`, `Ñ...` и другие следы битой перекодировки.
